3.5 HTTP Security Headers Analysis

The automated security header extraction tool reported an error: 'dict' object has no attribute 'startswith'. Therefore, a manual analysis of the provided HTTP headers for https://hcmute.edu.vn/ was performed.

Observed Security-Related Headers:

  • Set-Cookie: ASP.NET_SessionId=...; path=/; HttpOnly; SameSite=Lax
    • HttpOnly: Present. Good.
    • SameSite=Lax: Present. Good for most cases, provides CSRF protection.
    • Secure: Missing. This is a vulnerability.

Critically Missing HTTP Security Headers:

  • Strict-Transport-Security (HSTS): Not present. This header enforces HTTPS connections, protecting against SSL stripping attacks.
  • Content-Security-Policy (CSP): Not present. CSP helps prevent Cross-Site Scripting (XSS), clickjacking, and other code injection attacks by specifying allowed content sources.
  • X-Frame-Options: Not present. This header (or CSP frame-ancestors) protects against clickjacking attacks by controlling whether the site can be embedded in an <iframe>, <frame>, <embed> or <object>.
  • X-Content-Type-Options: Not present. Setting this header to nosniff prevents browsers from MIME-sniffing the content type, reducing risk from attacks that leverage content type confusion.
  • Referrer-Policy: Not present. Controls how much referrer information is sent with requests.
  • Permissions-Policy (formerly Feature-Policy): Not present. Allows fine-grained control over which browser features can be used by the site.

5. Recommendations

The following recommendations are prioritized to help improve the security posture of hcmute.edu.vn:

  1. [High Priority] Implement Comprehensive HTTP Security Headers:

    • Strict-Transport-Security (HSTS): Enforce HTTPS across the entire domain and subdomains. Example: Strict-Transport-Security: max-age=31536000; includeSubDomains; preload. Start with a smaller max-age for testing.
    • Content-Security-Policy (CSP): Implement a robust CSP to mitigate XSS and related attacks. This requires careful planning to define allowed sources for scripts, styles, images, etc. Start with a restrictive policy and gradually loosen as needed, or use Content-Security-Policy-Report-Only to test.
    • X-Frame-Options: Prevent clickjacking. Example: X-Frame-Options: SAMEORIGIN or DENY. Alternatively, use CSP frame-ancestors 'self'.
    • X-Content-Type-Options: Set to nosniff. Example: X-Content-Type-Options: nosniff.
    • Referrer-Policy: Control referrer information. Example: Referrer-Policy: strict-origin-when-cross-origin or no-referrer.
    • Permissions-Policy: Define which browser features the site can use. Example: Permissions-Policy: geolocation=(), microphone=(), camera=() (disabling these if not needed).

  2. [High Priority] Strengthen Email Security (SPF & DMARC):

    • Verify/Implement SPF Record: Create and publish an SPF TXT record that lists all authorized mail servers for hcmute.edu.vn. Given Google Workspace usage, this typically includes _spf.google.com. Example: v=spf1 include:_spf.google.com ~all (start with ~all (softfail) and move to -all (hardfail) after monitoring).
    • Implement DMARC Record: Publish a DMARC TXT record for _dmarc.hcmute.edu.vn. Start with a monitoring policy (p=none) and gradually move to p=quarantine and then p=reject. Example: v=DMARC1; p=none; rua=mailto:dmarc-reports@hcmute.edu.vn; ruf=mailto:dmarc-forensic@hcmute.edu.vn; fo=1; (adjust email addresses for reporting).

  3. [Medium Priority] Conduct Subdomain Audit and Secure Exposed Services:

    • Inventory all identified subdomains and determine their purpose and ownership.
    • Decommission any unnecessary or unmaintained subdomains.
    • Ensure consistent security configurations (HTTPS, security headers, authentication) across all active subdomains.
    • Pay special attention to subdomains like admin, cpanel, dev, gitlab, login, remote, staging, test, vpn. Apply strong authentication, access controls, and regular patching. Consider restricting access to these via IP whitelisting or VPN where appropriate.

  4. [Medium Priority] Minimize Information Disclosure:

    • Configure the web server (IIS) to remove or obfuscate the Server, X-AspNet-Version, and X-Powered-By headers from HTTP responses. This reduces the information available to attackers for targeted exploits.

  5. [Medium Priority] Secure Session Cookies:

    • Add the Secure flag to the ASP.NET_SessionId cookie (and all other session cookies) to ensure they are only transmitted over HTTPS. This requires the entire site to be served over HTTPS.
    • Evaluate if SameSite=Strict is appropriate for certain cookies for enhanced CSRF protection, though SameSite=Lax is a good default.

  6. [Medium Priority] Maintain Patch Management:

    • Ensure that Microsoft IIS 10.0 and ASP.NET (specifically version 4.0.30319 and the underlying .NET Framework) are kept up-to-date with the latest security patches. Regularly scan for known vulnerabilities associated with these technologies.

  7. [Low Priority] Investigate WHOIS Lookup Anomaly:

    • Determine the cause of the WHOIS lookup failure to ensure it's not indicative of a broader DNS resolution issue or a problem with the domain's registration status visibility.

  8. [Ongoing] Perform Regular Security Testing:

    • Conduct regular penetration testing and vulnerability scanning, especially focusing on web applications, identified subdomains, and authentication mechanisms.
    • Implement a continuous monitoring strategy for new subdomains and changes in existing services.

6. Attack Surface Evaluation

The reconnaissance data provides insights into the potential attack surface of hcmute.edu.vn:

  • Web Applications: The primary domain and numerous subdomains host web applications. The lack of key security headers (HSTS, CSP, XFO) makes these applications prime targets for XSS, clickjacking, and MITM (SSL stripping) attacks. The identified ASP.NET technology stack might have specific vulnerabilities if not properly patched and configured.
  • Email Infrastructure: With Google Workspace in use (DKIM present), the main concern is the apparent lack of SPF and DMARC records. This exposes the domain to email spoofing and phishing, which is particularly dangerous for an educational institution with many users.
  • Subdomain Proliferation: The 52 identified subdomains significantly expand the attack surface. Each subdomain could represent a distinct application, server, or service with its own set of potential vulnerabilities.
    • Services like admin.*, auth.*, cpanel.*, login.* are high-value targets for credential theft or unauthorized access.
    • dev.*, staging.*, test.* environments may have weaker security controls, debug features enabled, or contain sensitive data, making them attractive targets.
    • gitlab.* could expose source code, CI/CD pipelines, or internal developer credentials if not properly secured.
    • vpn.*, remote.* suggest remote access points that need robust security.
  • Information Disclosure: Server version information (IIS, ASP.NET) can help attackers tailor their exploits.
  • Entry Points for Further Testing:
    • Web application testing (OWASP Top 10) for the main site and high-risk subdomains.
    • Authentication mechanism testing on login, auth, portal subdomains.
    • Configuration review for cpanel, gitlab, vpn, remote.
    • Subdomain takeover checks, especially for less critical or potentially forgotten subdomains.
  • Social Engineering Potential: The lack of robust email security (SPF/DMARC) makes phishing campaigns targeting students, faculty, and staff more likely to succeed.
  • Security Posture vs. Industry Standards: The current configuration regarding HTTP security headers and email authentication (SPF/DMARC) falls short of common industry best practices. Educational institutions are frequent targets, and improving these areas is crucial.

Addressing the identified vulnerabilities, particularly implementing strong HTTP security headers and comprehensive email authentication, along with a thorough audit and securing of all subdomains, will significantly reduce the attack surface and improve the overall security posture of hcmute.edu.vn.
